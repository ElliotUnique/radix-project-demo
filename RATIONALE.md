
# Architectural Discussion
## warp-radix.cuh
`warpRadix` is a single-warp sort that emits the unique index that arranges each key in sorted order. This method was chosen over per-pass shuffling as the main dependency chain only involves low-latency ALU instructions; the ballots themselves are mutually independent and can be pipelined to take advantage of ILP (instruction-level parallelism). This can be done by manually rearranging the loop, though inspection of the SASS output indicates that the CUDA compiler (nvcc 11.8) can do so automatically when `start_bit` is known at compile-time. 

The ballot method also circumvents the need to build a histogram across radix values whose size would otherwise scale at a rate of `2^R`, where `R` is the bit-width of the radix. This removes the upper limit for radix width, allowing the sort to be resolved in a single sweep through the bits rather than processing smaller chunks sequentially.

If the sort is truncated before it fully resolves and the tie-break step omitted, the incomplete outputs instead form an encoding of the bucket structure seen in MSD-radix. This property is used in `block-radix.cuh`, where `narrowBucketMask`, a modified version of the `warpRadix` loop body, forms the warp-level core of `blockRadixKernel`.

## block-radix.cuh
### The Key-Shuffle
The multi-warp nature of `blockRadixKernel` precludes it from resolving by warp-level intrinsics alone, requiring bucket data to be communicated through shared memory and thus reintroducing the per-bucket histograms and multi-pass structure that `warpRadix` avoided (a pass is defined here by the advancement of the sorted-index variable). Since a worst-case distribution for `N` keys has `N / 2` buckets that each contain 2 keys in different warps, this results in a memory footprint `F` (in 32-bit words) that follows `F <= 2^R * (N / 2)`.

To improve this bound, I introduced a selective key-shuffling procedure that rearranges impure buckets into sub-bucket order (preserving initial ordering within sub-buckets). Since sub-buckets define the bucket structure of the next pass, this ensures that members of impure buckets are always held by contiguous threads/warps in `threadIdx.x` space, meaning impure buckets can only exist across neighbouring warp boundaries, restricting the number of impure buckets to `<= W - 1`, where `W` is the total number of warps.

However, maintaining stability across the shuffle requires each participating key to know how many keys from lower warps belong in the same sub-bucket. To preserve this information, each warp submits its own part of the histogram data (specifically a prefix sum across the histogram elements) to separate locations to be consumed piecewise. Since a bucket spanning `k` warps consumes `k - 1` boundaries, summing across all impure buckets gives a warp participation of `<= 2 * (W - 1)`. With the sort allocating 32 keys per warp, this yields an upper bound of `F <= 2^R * 2 * ((N / 32) - 1)`; I then chose `R = 4` to give `F <= N - 32`, making the entire cross-warp communication footprint no greater than the original key array itself.

This process has a computational cost and requires additional `__syncthreads` gates, all of which increase the latency of each pass. However, reducing the number of impure buckets necessarily increases the number of *pure* buckets, and these can be resolved via warp intrinsics and register operations by the same mechanism as `warpRadix`. By accelerating the breakdown of impure buckets into pure ones, the average number of passes that require expensive memory transactions decreases, recouping the key-shuffle's cost, though to what extent is an open question.

One factor that affects this is the distribution of the keys. Distributions with high entropy tend to divide quickly into smaller buckets, allowing warps to drop out of cross-warp communications early, whereas duplicate-heavy distributions result in persistent impure buckets, which trigger key-shuffles on every pass.

In addition to these entropic extremes, I validated the algorithm against distributions with intermediate entropy levels as well as pre-sorted and reverse-sorted keysets. I also performed tests for boundary values of N (0, 1, 1024) and non-warp-aligned N (&plusmn;1, +16, random).

### The Messenger System
There is an inherent work imbalance between pure and impure threads, as impure threads must participate in cross-warp communication and key-shuffle procedures that pure threads do not require. On top of this, impure buckets shrink over time and are seldom split evenly between warps, so if impure threads were solely responsible for their own bucket's cross-warp data, this workload would have to be dynamically allocated across an ever-changing thread pool. These reallocations would incur their own computational costs, and small thread pools would need to stack their work to ensure the bucket's full data was properly represented.

To address this, I created a "messenger" system where each thread in an impure warp is given responsibility over a single histogram prefix sum element from an impure bucket, regardless of its own bucket membership. The earlier choice of `R = 4` gives 16 elements per bucket histogram and combines with the guarantee of no more than two impure buckets per warp, allowing each warp to allocate half its lanes to service each impure bucket; lanes 0-15 construct, submit and retrieve data for the bucket that straddles the lower boundary, and lanes 16-31 do so for the upper-boundary bucket.

This entirely replaces the dynamic work allocation with a static system assigned at compile time that allows pure threads to help rebalance the workload instead of waiting for the impure threads to catch up.

### Index Anchoring
The key-shuffle phase redistributes keys within each impure bucket in sub-bucket order first and relative `threadIdx.x` order second. Since the key-shuffles of different buckets cannot be mixed and the contiguity of impure buckets must be maintained, threads must all agree on a suitable anchor location in shared memory to scatter their keys relative to. Early in development, this relied on the lowest thread in the bucket bit-packing its `threadIdx.x` value into one of its warp's histogram prefix sum elements. This meant the thread had to identify itself to its messengers, and required all other warps' messenger groups to carefully handle their arithmetic so as not to mangle the encoding or corrupt the prefix sum data.

Searching for a more robust replacement, I observed that the `bucket_base` variable (under a different name at the time) directly encoded the identity of each key's bucket, and was thus homogeneous across any given bucket. Additionally, by maintaining the contiguity of impure buckets, I knew by induction that the intervals they occupied in `threadIdx.x` space would begin and end in exactly the same locations they occupy in sorted position: in the zeroth iteration the entire array constitutes an impure bucket and is rearranged in contiguous order, then each subsequent pass maintains this pattern while impure buckets occupy progressively smaller subsets of the sort.

This meant that `bucket_base` was always equal to the `threadIdx.x` value of the first thread in the impure bucket, which was exactly the value the bit-packing system was generating, but as a raw value unique to each bucket and shared by all its members. Adopting `bucket_base` as the direct anchor value therefore ended the bit-packing reliance and the computational work-arounds it entailed.

This identity is used both in the key-shuffle itself and the concurrent metadata reconstruction process in which impure threads compute their new key's metadata in advance of its arrival.
